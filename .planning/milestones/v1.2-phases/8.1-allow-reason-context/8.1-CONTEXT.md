# Phase 8.1: Allow-reason context - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

When a track passes with mild profanity (severity=1, below the skip threshold), the dashboard
now-playing card shows an additional amber "Mild language" badge alongside the existing green
"Passed" badge. Tracks with severity=0 show only "Passed" unchanged.

Backend changes: add `severity` to `eval_result` SSE events and `now_playing.json` so the
frontend can render the additional badge. The daemon already computes `severity` — it just
needs to be propagated into the event payloads.

Does NOT cover: drug reference or sexual content detection (v1.3), per-category toggle UI
(v2+), or changes to the skip-reason badges in the incident log feed.
</domain>

<decisions>
## Implementation Decisions

### Badge Architecture (Multi-badge model)

- **D-01:** Keep the existing green "Passed" badge for all passing tracks. When severity=1,
  render an ADDITIONAL amber "Mild language" badge next to it. This is NOT a replacement —
  both badges show simultaneously.
  - severity=0 → green "Passed" only
  - severity=1 → green "Passed" + amber "Mild language"
  - This sets the groundwork for v1.3 where a track can meet multiple criteria (language,
    drug reference, sexual content) and each criterion gets its own badge.

- **D-02:** The badge area in the now-playing card transitions from a single `<span>` to a
  badge container `<div>` that can hold multiple badges side by side (flex row). The eval_state
  badge (Passed/Evaluating/Skipped/etc.) is always first; criteria badges follow.

- **D-03:** "Mild language" badge CSS: amber/yellow color using new `.badge--mild-language`
  class. Reuse the amber color already used in `.badge--evaluating` for consistency, or
  choose a distinct amber shade — implementer's discretion.

### Backend Data Flow

- **D-04:** Add `severity` (integer 0–3) to the `eval_result` SSE event payload in `daemon.py`.
  The variable is already computed at line 248 (`action, reason, severity = await
  content_checker.check(track)`). Only `eval_result` events need `severity`; `track_change`
  events do not (severity is unknown at that point).

- **D-05:** Add `severity` to `now_playing.json` writes in `daemon.py`. The now_playing.json
  write happens after evaluation — severity is available at that point. `GET /now-playing`
  in `web_ui/main.py` passes the file through as-is, so no web_ui change is needed if
  severity is in the file.

- **D-06:** For non-profanity eval outcomes (explicit flag skip, no-lyrics, fsm-off, paused):
  severity in those cases is either 3 (explicit skip computed) or 0 (no profanity scan ran).
  Include `severity` in all `eval_result` events for consistency; the frontend only acts on
  it when `eval_state === "passed"`.

### Frontend Rendering

- **D-07:** The `eval_result` event handler and hydration function receive `severity` from
  the payload. When `eval_state === "passed"` AND `severity >= 1`, render the
  `.badge--mild-language` badge after the eval_state badge.

- **D-08:** `currentTrackId` guard (NOW-07, from Phase 8 D-07) is unchanged. The severity
  badge rendering happens inside the same guarded block as the eval_state update.

- **D-09:** No severity badge is rendered for skipped/no-lyrics/fsm-off/paused states —
  the eval_state badge already communicates the outcome. Severity badge only applies to
  "passed" outcomes.

### Claude's Discretion

- Exact amber hex value for `.badge--mild-language` (can match `.badge--evaluating` amber
  or use a slightly different hue to distinguish "active mild" from "transient evaluating")
- Whether badge container `<div>` gets a CSS class like `.badge-group` or just uses inline
  flex style
- Badge label for "Mild language" — "Mild language" is the confirmed text
- How to handle severity > 1 that still passes (not currently possible given default
  PROFANITY_MIN_SEVERITY=2, but safe to only render badge for severity=1 for now)
</decisions>

<specifics>
## Specific Ideas

- Multi-badge model is deliberate architectural groundwork for v1.3: drug reference and
  sexual content detection will each get their own badge in the same container
- "Mild language" is the exact confirmed badge text
- "Passed" badge stays green and unchanged — severity badge is additive, not a replacement
- The badge container approach (D-02) makes v1.3 badge additions a one-liner change in the
  rendering logic
</specifics>
