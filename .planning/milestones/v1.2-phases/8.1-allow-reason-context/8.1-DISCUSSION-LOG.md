## Gray Areas Identified

### From codebase analysis
- `content_checker.check()` returns `(action, reason, severity)` — severity already computed
- `daemon.py` captures `severity` at line 248 but does NOT include it in `eval_result` events or `now_playing.json`
- Phase 8 badge system uses a single `<span>` for eval_state — single badge model

### Gray area 1: Badge presentation
**Question:** What secondary context should appear with "Mild language" badge?
**Options presented:** Badge text only / Browser tooltip / Sub-label text below badge

**User response:** Additional badge that reads "Mild Language" when severity=1, while still
showing the green "Passed" badge. Explicitly mentioned this should "set the groundwork for
the future state where a song can meet multiple criteria (language, drug reference, sexual
content)."

**Decision captured:** Multi-badge architecture — keep "Passed" badge, add amber "Mild
language" badge alongside it. Badge container replaces single `<span>` to support multiple
simultaneous badges.

### Gray area 2: Badge color
**Question:** Should "Mild language" be amber (distinct from Passed) or green (same as Passed)?

**User response:** Amber/yellow (Recommended)

**Decision captured:** `.badge--mild-language` uses amber/yellow color, visually distinct
from green "Passed".

## Key Architectural Decision

The user explicitly reframed this from a badge replacement (as described in ROADMAP.md) to
a badge addition. This multi-badge model is intentional forward-thinking for v1.3 where
drug reference and sexual content will each get their own badge.
