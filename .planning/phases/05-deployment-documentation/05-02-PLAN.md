H1 title: "Proxmox / LXC Multicast Notes"
Opening sentence: "Why Sonos SSDP discovery may fail in LXC containers, and what to do about it."

H2 "Why This Matters"
Explain:
- Sonos speakers are discovered via SSDP (Simple Service Discovery Protocol), which uses UDP multicast on port 1900.
- When the daemon runs inside a Proxmox LXC container, it uses network_mode: host — sharing the host's network stack.
- Multicast traffic is not always forwarded across the Linux bridge (typically vmbr0) into LXC network namespaces by default.
- Show the log message the daemon produces when SSDP fails:
    [SONOS] No speakers found via SSDP. Check firewall (allow UDP 1900 multicast) and Proxmox bridge multicast forwarding. Set SONOS_SPEAKER_IPS as a fallback.

H2 "Fix: Enable Multicast Bridge Forwarding"
Explain:
- Enable multicast forwarding on the Proxmox host bridge. The exact steps depend on your Proxmox version and bridge configuration.
- Link to official Proxmox documentation: https://pve.proxmox.com/wiki/Network_Configuration
Do NOT include specific nftables or iptables commands.

H2 "Escape Hatch: SONOS_SPEAKER_IPS"
Explain:
- If configuring bridge multicast is not possible, bypass SSDP entirely by setting speaker IPs directly in .env.
- Show the exact env var format (matching .env.example):
    SONOS_SPEAKER_IPS=Dining Room=192.168.1.50,Living Room=192.168.1.51
- Explain: use the exact room names as Spotify reports them. Find speaker IPs in the Sonos app: Settings -> System -> [Room] -> About [Room]
- State: when SONOS_SPEAKER_IPS is set, SSDP discovery is skipped entirely.
