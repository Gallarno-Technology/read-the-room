<task type="auto">
  <name>Task 1: Fix Makefile setup guard to handle root-owned directory</name>
  <files>Makefile</files>
  <action>
    Current line 7:
      `@[ ! -f lyrics_cache.db ] || sudo rm -f lyrics_cache.db`

    This guard uses `-f` (true only for regular files). When Docker has created lyrics_cache.db as a root-owned directory, `-f` is false, the sudo rm is skipped, and `touch` fails with "Permission denied".

    Replace lines 7-8 with a three-branch compound guard that covers all cases:
      - `-d lyrics_cache.db` → directory exists (root-owned from bad Docker run): `sudo rm -rf lyrics_cache.db`
      - `-f lyrics_cache.db` → regular file exists (already set up): `sudo rm -f lyrics_cache.db`
      - neither → nothing to remove

    Write it as two separate guard lines before the touch, keeping the existing Makefile style:
    ```
    @[ ! -d lyrics_cache.db ] || sudo rm -rf lyrics_cache.db
    @[ ! -f lyrics_cache.db ] || sudo rm -f lyrics_cache.db
    @touch lyrics_cache.db
    ```

    The `-d` check runs first, removing the directory case. The `-f` check then handles any lingering regular file. Then `touch` creates a fresh empty file in either case.

    Do not change any other lines in the Makefile.
  </action>
  <verify>
    <automated>grep -n "sudo rm -rf" /home/cgallarno/Development/spotify-sentiment/Makefile</automated>
  </verify>
  <done>Makefile contains `sudo rm -rf lyrics_cache.db` in the setup target (the -d directory guard). Both the -d and -f guard lines appear before `touch lyrics_cache.db`.</done>
</task>
