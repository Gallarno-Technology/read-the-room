  <action>
Add `user: "${UID}:${GID}"` to the `daemon` service block in docker-compose.yml, immediately after the `env_file` line. The resulting service block must look like:

```yaml
services:
  daemon:
    build: .
    restart: always
    network_mode: host
    env_file: .env

    volumes:
      ...
```

This ensures the container process runs as the host user's UID/GID. Any bind-mount files the container creates (lyrics_cache.db, state.json) will be owned by the host user, not root. Docker Compose expands ${UID} and ${GID} from the shell environment at `docker compose up` time.

  <name>Task 2: Fix make setup — defensively remove root-owned lyrics_cache.db before touch</name>
  <files>Makefile</files>

  <read_first>Makefile — read the full setup target before editing</read_first>

  <action>
Replace the `touch lyrics_cache.db` line in the `setup` target with a two-step sequence that handles a pre-existing root-owned file:

```makefile
setup:
	@[ -f state.json ] || echo '{"last_track_id": null}' > state.json
	@mkdir -p token_cache
	@[ ! -f lyrics_cache.db ] || sudo rm -f lyrics_cache.db
	@touch lyrics_cache.db
	@[ -f .env ] || cp .env.example .env
	@echo "Setup complete. Edit .env with your Spotify credentials, then run: make auth"
```

Explanation of change:
- `[ ! -f lyrics_cache.db ] || sudo rm -f lyrics_cache.db` — if the file exists (possibly root-owned from a previous run), remove it with sudo so the host user can recreate it. If it does not exist, the condition short-circuits and nothing happens.
- `touch lyrics_cache.db` — creates a fresh, host-user-owned empty file (which Docker will then bind-mount).

This is idempotent: first run creates the file; subsequent runs delete and recreate it (harmless since it is a cache). The sudo rm is safe because it only targets this specific file in the project directory.

Do NOT change any other target in the Makefile.
  </action>

On line 105 of daemon.py, change:

```python
result = sp.currently_playing()
```

to:

```python
result = sp.current_playback()
```

That is the ONLY change. Do not touch any other line.

Why this works: `GET /me/player/currently-playing` (currently_playing) returns a subset of the playback state that omits the `device` object. `GET /me/player` (current_playback) returns the full playback context including `device.is_restricted`. The `item` key is present in both responses, so all downstream track-detection logic (track_id comparison, explicit flag, content_checker.check()) is unaffected. After this fix, `result.get("device", {})` on lines 134–136 will return the real device object, and `is_restricted` will be True for Sonos speakers, routing skips through SocoSkipClient.

Do NOT change the module docstring (which mentions /me/player/currently-playing — that is historical context, not a routing decision) unless it causes confusion.
