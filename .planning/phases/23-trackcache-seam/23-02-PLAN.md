    Step 3 — Refactor ContentChecker.check() to single-exit-point pattern (D-06 + research recommendation):

    The current check() has 5 return points. Extract all pipeline logic into a private
    `async def _run_pipeline(self, track: dict) -> TrackEvalResult:` method that contains
    the unchanged pipeline body verbatim (Tier 1 explicit check through no_lyrics_service fallback).

    Then replace `check()` body with:
    ```python
    async def check(self, track: dict) -> "TrackEvalResult":
        # Cache fast-path (D-06 step 1)
        if self.track_cache is not None:
            cached = await self.track_cache.get(track["id"])
            if cached is not None:
                return cached

        # Run the full pipeline
        result = await self._run_pipeline(track)

        # Cache write after pipeline completes (D-06 step 3)
        if self.track_cache is not None:
            await self.track_cache.put(track["id"], result)
        return result
    ```

    The `_run_pipeline` method:
    - Contains the entire previous body of `check()` exactly as-is — do not change any scan
      logic, logging, or return values.
    - All 5 existing return statements remain inside `_run_pipeline`; none are in `check()`.
    - Public API of `check()` is unchanged for all existing callers.

    IMPORTANT: The cache fast-path runs BEFORE Tier 1 (explicit check). This is correct per
    D-06: a cached "skip/explicit" result is returned immediately without re-evaluating Tier 1.
    This does NOT violate the no-short-circuit rule — that rule applies within the scan pipeline
    itself (Tiers 3-5), not to this higher-level cache.

<threat_model>
## Threat Model (ASVS Level 1)

| Threat | Vector | Mitigation | Severity |
|--------|--------|------------|----------|
| Cache poisoning via track_id collision | Two tracks sharing a Spotify track ID | Spotify track IDs are globally unique and immutable; no two tracks share an ID | LOW |
| Import cycle causing startup crash | content_checker.py imports TrackCache from track_cache.py; track_cache.py imports TrackEvalResult from content_checker.py | `from __future__ import annotations` + `TYPE_CHECKING` guard in content_checker.py breaks the runtime cycle entirely | HIGH → BLOCKED |
| Profile-change silently disabling cache | _build_content_checker profile-change call site not updated | Both call sites are explicitly updated in Task 2; Pitfall 4 from research explicitly addressed | MEDIUM → BLOCKED |
| Cache returning stale result after track re-release | Same track_id, different content (theoretical) | Spotify track IDs are immutable per released track; content never changes after release | LOW |

**High-severity threats:** Import cycle is the only high-severity threat; it is blocked by the TYPE_CHECKING guard. No additional controls required.
</threat_model>
